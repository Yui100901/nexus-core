# nexus-core 需求推进清单

本文档用于跟踪 `nexus-core` 后续重构和功能实现进度。

状态说明：

- `[ ]` 待处理。
- `[~]` 部分完成，需要继续整理或补齐。
- `[x]` 已完成。

## 0. 当前阻塞项

这些问题会影响项目编译、启动或核心链路验证，应优先处理。

- [x] 移除旧的 `nexus-core/sc` 包依赖。
  - 仓储层已统一为标准 `context.Context`。
- [x] 修复数据库初始化链路。
  - 当前 `main.go` 调用了 `base.MainDBManager`、`base.InitDBManager`、`GetDefaultDB`。
  - 当前 `persistence/base` 中没有这些定义。
  - 需要统一为单库初始化，或实现多数据库管理器。
- [x] 初始化 `global.DB`。
  - 当前大量 service 直接使用 `global.DB`。
  - 启动时需要明确完成赋值，并处理初始化失败。
- [x] 修复 `go test ./...` 编译失败。
  - 完成上述阻塞项后，需要保证全项目至少能编译通过。
- [x] 清理旧架构残留代码。
  - 删除不可用的旧方法，或将其迁移到新上下文和新模型。
  - 避免同一业务同时存在旧实现和新实现。

## 1. 架构与边界重构

### 1.1 分层边界

- [x] 明确 Controller、Service、Repository、Model、Entity、DTO 的职责。
- [x] Controller 只负责参数绑定、错误映射和响应输出。
- [x] Service 负责业务规则、事务编排和领域对象调用。
- [x] Repository 只负责数据库读写。
- [x] Model 只表达数据库结构。
- [x] Entity 只表达领域行为和业务状态。
- [x] DTO 只表达接口输入输出。

### 1.2 上下文与事务

- [ ] 所有数据库调用支持传入 `context.Context`。
- [ ] 需要事务的业务统一在 Service 层开启事务。
- [ ] 事务内禁止绕回 `global.DB` 进行写操作。
- [ ] 注册、绑定、激活 License 需要放入同一个事务。
- [ ] 删除产品、删除 License、删除节点等关联删除需要事务保护。

### 1.3 错误处理

- [ ] 统一业务错误类型。
- [ ] 区分参数错误、资源不存在、状态冲突、权限不足和内部错误。
- [ ] Controller 根据业务错误映射 HTTP 状态码。
- [ ] 修正目前大量 `InternalError` 承载业务错误的问题。
- [ ] 修正错误文案，例如节点查询失败不应返回 `product not found`。

### 1.4 命名与接口风格

- [ ] 统一 REST 路由命名。
  - 例如 `/license/createLicense` 可整理为 `POST /licenses`。
  - 例如 `/node/getByID` 可整理为 `GET /nodes/:id`。
- [ ] 统一 JSON 字段命名风格。
  - 建议统一使用 snake_case。
- [ ] 统一状态字段命名。
  - 节点绑定状态建议统一为 `status` 或 `binding_status`。
- [ ] 修复 Swagger 注释和实际路由不一致的问题。
- [ ] 清理空接口或未实现接口。

### 2.4 节点绑定模型

- [~] 节点与 License 绑定表。

### 2.5 节点控制模型

- [ ] 新增控制服务定义表。
