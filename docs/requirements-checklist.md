# nexus-core 需求推进清单

本文档用于跟踪 `nexus-core` 后续重构和功能实现进度。当前项目已经完成可运行状态、License 主链路、限制与状态校验，下一步重点转向节点控制能力。

状态说明：

- `[ ]` 待处理。
- `[~]` 部分完成，需要继续整理或补齐。
- `[x]` 已完成。

优先级说明：

- `P0` 当前主线，优先实现。
- `P1` 主线配套，影响可用性、可验证性或文档交付。
- `P2` 功能补齐，重要但不阻塞下一阶段主线。
- `P3` 长期完善，偏监控、运营、体验和工程质量。

## 1. 当前优先级评估

### P0：实现节点控制最小闭环

目标是先跑通一条可验证链路：

`定义控制服务 -> 节点上报能力 -> 服务端创建指令 -> Schema 转换 -> 下发到节点 -> 回执结果 -> 查询结果`

优先建议按以下顺序推进：

1. 控制服务定义与产品授权关系。
2. 节点能力上报与保存。
3. Schema 转换与校验。
4. 控制指令创建与状态流转。
5. 先支持一种下发方式，建议 HTTP 或 WebSocket 二选一。
6. 指令执行结果回执与查询。

### P1：补齐控制链路的可验证性

- 补充控制服务、节点能力、Schema 转换、指令执行的测试。
- 增加注册、心跳、节点控制接口示例。
- 修复并重新生成 Swagger 文档。
- 提供最小 Demo 数据或调用脚本。

### P2：收敛核心管理能力

- 产品、License、节点仍有一些管理类接口未完整收敛。
- 当前不阻塞节点控制主线，但会影响后续后台管理体验。
- 建议等控制闭环跑通后再统一整理。

### P3：监控、审计和运营能力

- 在线状态恢复、统计查询、事件日志、审计日志属于运营层能力。
- 当前已有内存在线状态和并发统计基础，但还没有完整查询与持久化。
